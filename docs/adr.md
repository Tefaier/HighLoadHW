# ADR-001. Архитектурный стиль

## Контекст
1) Высокий объём чтения
2) Требуется независимое масштабирование горячих контуров
3) Отдельные пользовательские сценарии для клиента, курьера и ресторана 
4) Требуется высокая доступность и сохранность данных для критических операций заказа и оплаты
5) Много доменов: каталог, меню, корзину, заказ, оплату, доставку, уведомления, кабинет ресторана

Важно минимизировать связанность между доменами и дать возможность масштабировать их независимо. Особенно это критично для каталога и меню, которые нагружены существенно сильнее, чем интерфейс ресторана или уведомления

## Альтернативы

### 1. Модульный монолит

**Плюсы:**  
Простая разработка на старте; одна кодовая база; единое окружение; легче трассировать вызовы внутри процесса; нет сетевых накладных расходов между модулями; проще тестировать end-to-end на раннем этапе

**Минусы:**  
Хотя модули можно логически разделить, физически они остаются в одном процессе. Это приводит к тому, что все горячие участки системы масштабируются вместе, даже если реальная нагрузка сосредоточена только на каталоге и меню. С ростом системы начинает расти связанность между модулями, а изменение одного домена часто затрагивает соседние. Отказ в одном модуле повлияет на весь процесс.
### 2. Микросервисы с разбиением системы по бизнес-доменам

**Плюсы:**  

Микросервисы дают возможность:

- масштабировать только те сервисы, которые действительно нагружены;
- изолировать сбои и ограничивать blast radius;
- независимо развивать домены разными командами;
- внедрять асинхронные сценарии для длительных операций;
- использовать разные модели хранения и кэширования там, где это оправдано.

**Минусы:**  
- выше операционная сложность;
- сложнее согласованность данных между сервисами

## Решение

Для нашей системы с 20 млн MAU и пиковыми 2000+ RPS модульный монолит будет слишком жёстко связан с общим жизненным циклом деплоя и масштабирования, поэтому решено использовать микросервисную архитектуру с разбиением системы по бизнес-доменам
## Обоснование

С точки зрения НФТ микросервисы покрывают (с учетом озвученных выше плюсов):

- **Throughput** — за счёт горизонтального масштабирования по доменам;
- **Durability** — за счёт изоляции критических транзакционных операций;
- **Scalability** — за счёт независимого масштабирования компонентов без изменения кода
## Последствия

Положительные:

- независимое масштабирование;
- лучшее разделение ответственности;
- более устойчивое поведение при пиках;
- проще строить разные контуры для клиента, курьера и ресторана

Отрицательные:

- выше операционная сложность;
- сложнее согласованность данных между сервисами

# ADR-002. Базы данных

## Контекст

Система хранит смешанный набор данных: транзакционные сущности заказа и оплаты, справочники ресторанов и меню, рейтинги и отзывы, а также большие медиаобъекты — фотографии блюд. Для заказов критичны целостность, транзакционность и гарантия сохранения. Для изображений критичны дешёвое хранение, масштабирование и быстрая отдача

## Альтернативы

### 1. NoSQL-документная БД как основное хранилище

**Плюсы:** 
Гибкая схема; удобна для быстро меняющихся документов; может хорошо масштабироваться по записи; удобно хранить вложенные структуры без нормализации.

**Минусы:**  
Система доставки еды имеет транзакционный контур, где важны строгие гарантии: создать заказ, сохранить оплату, зафиксировать статусы, избежать дублей. В NoSQL сложнее обеспечить ACID-поведение и корректную согласованность между сущностями. Сложные фильтры по ресторану, кухне, рейтингу и цене, а также связи между заказом, рестораном и пользователем, обычно становятся менее удобными. При росте аналитических и операционных запросов возникает необходимость строить дополнительные проекции и костыли
### 2. Хранение изображений внутри БД как BLOB

**Плюсы:**  
Единое хранилище; проще логически в начале; транзакционно можно связывать метаданные и бинарные данные.

**Минусы:**  
База резко разрастается; ухудшается скорость бэкапов и восстановления; возрастает нагрузка на транзакционные операции; репликация становится тяжелее; кэширование и CDN использовать неудобно; обслуживание системы становится дороже

### 3. Файловая система или NFS для изображений

**Плюсы:**  
Просто на старте; не требует отдельного object storage; легко понять модель хранения.

**Минусы:**  
Плохая горизонтальная масштабируемость; сложная синхронизация между инстансами; риск рассинхронизации; сложность отказоустойчивости; неудобно строить CDN; бэкапы и восстановление становятся неудобными

### 4. PostgreSQL + S3

Плюсы:
PostgreSQL хорошо подходит для:

- заказа как транзакционной сущности;
- строгой целостности;
- сложной фильтрации и сортировки;
- индексов по кухне, цене, рейтингу, статусу;
- реализации ACID-транзакций

S3 подходит для:

- фото блюд и ресторанов;
- вложений и других медиа;
- дешёвого и масштабируемого хранения;
- доставки контента через CDN без нагрузки на БД

Минусы:
- S3 сложно поднимать
- Не сможем реализовать такое же подробное описание для food items как если бы использовали условный MongoDB
## Решение

Использовать:

- **PostgreSQL** как основную реляционную СУБД для доменных и транзакционных данных;
- **S3-compatible object storage** для фото

## Обоснование

Для распределённой системы с несколькими регионами и высокой нагрузкой Файловая система или NFS это плохое решение: меньше отказоустойчивость, сложнее масштабировать.

Заказ и оплата для нас критически важны. Возможность добавления любых полей в food items не очень полезна. Поэтому основой должна быть реляционная модель, а не документная.

Если хранить изображения как Blob в PostgreSQL, это ухудшит производительность

Отсюда PostgreSQL + S3 - более хорошее решение.
## Последствия

Положительные:

- строгая целостность заказов;
- удобная фильтрация и выборки;
- предсказуемая эксплуатация;
- медиа отделены от транзакционного слоя;
- проще реализовать бэкапы и политики хранения.

Отрицательные:

- потребуется внимательно проектировать индексы и миграции;
- при росте объёма заказов могут понадобиться партиционирование и реплики;
- отдельная система object storage требует политики доступа и CDN-интеграции

# ADR-003. Паттерны коммуникации

## Контекст

В системе есть два типа коммуникации:

1. короткие запросы с немедленным ответом, например просмотр меню или создание корзины;
2. длинные и ненадёжные по времени процессы, например подтверждение платежа, уведомления, назначение курьера, обновление статусов от внешних систем.

Если строить всё только на синхронных вызовах, то цепочка заказа станет слишком длинной, latency вырастет, а устойчивость упадёт. Если строить всё только на асинхронщине, ухудшится UX и станет сложнее давать немедленный ответ пользователю.

## Альтернативы

### 1. Только синхронные REST-вызовы между всеми сервисами

**Плюсы:**  
Простая модель взаимодействия; легко понять и отлаживать; предсказуемый request/response; не нужен брокер сообщений; меньше дополнительных компонентов инфраструктуры

**Минусы:**  
Это создаёт длинную цепочку зависимостей: заказ зависит от ресторана, ресторан зависит от платежа, платеж зависит от внешнего провайдера, уведомление зависит от всего остального. Любой сбой или замедление удлиняет p99 и повышает вероятность каскадных ошибок. Система становится чувствительной к latency внешних интеграций. При росте нагрузки синхронные блокировки и таймауты быстро ухудшают UX

### 2. Только асинхронная модель на событиях

**Плюсы:**  
Очень хороша для масштабирования; слабая связанность; устойчивость к пикам; легко подключать новые consumer-ы; лучше переносит временные сбои зависимостей.

**Минусы:**  
Пользователь не получает немедленного результата; сложнее обеспечивать UX; повышается сложность отладки; возникает eventual consistency даже там, где она нежелательна. Для таких операций, как просмотр меню или быстрая валидация корзины, pure async выглядит неоправданно тяжёлым. Кроме того, некоторые сценарии требуют немедленного ответа о валидности данных, например наличие блюда или доступность ресторана

### 3. gRPC как единый универсальный протокол

**Плюсы:**  
Высокая производительность; строгие контракты; удобен для внутреннего взаимодействия; хорошо подходит для low-latency service-to-service вызовов.

**Минусы:**  
Для внешних клиентов и публичных API gRPC менее удобен, чем REST; сложнее в отладке через браузер и внешние инструменты; не всегда хорошо подходит для сценариев с широкой интеграцией разных платформ; не решает проблему длительных процессов сам по себе. Если использовать его повсеместно, появится лишняя жёсткость интерфейсов и усложнение интеграционного слоя

### 4. Polling вместо событий и webhooks

**Плюсы:**  
Просто реализовать; удобно, если внешняя система не умеет пушить события; легко контролировать частоту обращений.

**Минусы:**  
Неэффективно по нагрузке; увеличивает latency обновления статусов; создаёт лишние запросы; плохо масштабируется при большом количестве заказов; усложняет актуализацию статусов в реальном времени

## Решение

Использовать **гибридную схему коммуникации**:

- **REST/JSON через API Gateway** для внешних клиентов;
- **синхронные вызовы** внутри системы там, где нужен быстрый результат;
- **асинхронные события через message broker** для изменений состояния, уведомлений и фоновых процессов;
- **webhooks** для внешних интеграций, в частности платежей;
- **outbox + idempotency + saga** для надёжной обработки критических бизнес-процессов.

## Обоснование

Чистый async ухудшит пользовательский опыт и усложнит реализацию простых read-path сценариев
Только синхронные запросы не позволят достичь высокой доступности системы
gRPC можно применять точечно внутри периметра, но он не должен быть единственным паттерном коммуникации. Для внешнего API и публичных мобильных клиентов REST/JSON практичнее
Система должна давать актуальный статус заказа и быстро обновлять клиента и курьера. Polling для этого слишком дорог и медленен

Синхронный слой нужен там, где пользователь ожидает быстрый ответ: каталог, меню, корзина, базовая валидация заказа. Асинхронный слой нужен там, где важнее надёжность и масштабируемость, чем немедленный финальный ответ: платежи, уведомления, синхронизация статусов, курьерские события.

Гибридная схема соответствует требованиям по:

- **Latency** — сокращается критический путь;
- **Throughput** — часть нагрузки уходит в очередь;
- **Availability** — сбои внешних систем не ломают весь пользовательский путь;
- **Durability** — запись заказа и событий может быть отделена и подтверждена надёжно.
## Последствия

Положительные:

- стабильная latency на пользовательском пути;
- независимое масштабирование фоновых обработчиков;
- лучшее поведение при сбоях внешних интеграций;
- высокая надёжность доставки событий;
- удобно строить уведомления и обновление статусов

Отрицательные:

- неизбежна eventual consistency в части процессов;
- нужны отдельные механизмы надёжности сообщений;
- появляется больше инфраструктурных компонентов;
- усложняется трассировка и отладка
